Version: 1.0

1. EXECUTIVE SUMMARY
1.1 Purpose
This User Requirements Document defines the functional and non-functional requirements for the FinCore Platform, a comprehensive financial services platform designed to manage currency exchange, cash collection, customer onboarding, and transaction processing operations.
1.2 Scope
The FinCore Platform consists of two primary user interfaces:
Client Section: Self-service portal for customers to manage beneficiaries, initiate transactions, and perform currency exchanges
Admin Section: Back-office system for staff to process transactions, manage customers, ensure compliance, and oversee financial operations
1.3 Project Objectives
Streamline customer onboarding through digital application processing
Enable efficient cash collection and currency exchange operations
Ensure regulatory compliance through integrated AML/KYC verification
Provide comprehensive financial transaction management and reporting
Maintain accurate accounting through integrated General Ledger system

2. STAKEHOLDERS
2.1 Primary Users
End Customers: Individuals and businesses requiring financial services
Administrative Staff: Back-office personnel managing operations
Compliance Officers: Staff responsible for AML/KYC verification
Accountants: Finance team managing ledgers and reconciliation

Management: Executives requiring reports and oversight

The FinCore Platform serves as a central hub for:

5. ADMIN SECTION - FUNCTIONAL REQUIREMENTS
5.1 Cash Collection Module (Admin)
5.1.1 Business Requirements
Provide administrative staff with tools to manage, dispatch, and track cash collection requests from customers.
5.1.2 Functional Requirements
FR-ACC-001: Collection Request Management
System shall display dashboard of all cash collection requests
System shall show requests grouped by status: New, Assigned, In Progress, Completed, Cancelled
System shall display request details: customer name, amount, address, requested date/time
FR-ACC-002: Collection Processing
System shall allow admin to update collection status in real-time
System shall capture:
Actual collection amount
Currency breakdown (if multiple denominations)
Collection time and location
Agent signature and customer signature

System shall verify collected amount matches requested amount

5.2 Incoming Transactions Module
5.2.1 Business Requirements
Enable admin staff to receive, process, and reconcile incoming payments from clients 
5.2.2 Functional Requirements
FR-IT-001: Transaction Reception
System shall receive incoming transaction notifications for all incoming transactions:
System shall parse transaction files automatically
System shall extract: sender details, amount, currency, reference, date
FR-IT-002: Reconciliation
System shall mark matched and approved transactions as "Reconciled"
System shall post transactions to customer accounts
System shall generate GL entries automatically
System shall send notification to customers about credited funds
System shall maintain audit trail of all transaction changes
FR-IT-003: Exception Handling
System shall flag suspicious transactions for review:
Sender is on sanctions list
Receiver is on sanctions list
System shall allow admin to reject transactions with reason

5.7 AML & KYC Module
5.7.1 Business Requirements
Ensure regulatory compliance through comprehensive Anti-Money Laundering (AML) and Know Your Customer (KYC) processes, including sanctions screening, transaction monitoring, and suspicious activity reporting.
5.7.2 Functional Requirements
FR-AML-001: KYC Verification
System shall integrate with third-party KYC providers (Sumsub)
System shall perform identity verification:
Document authenticity check
Liveness detection (selfie verification)
Facial recognition match with ID photo
Data extraction from documents (OCR)
System shall return verification result: Passed, Failed, Manual Review Required
System shall store verification evidence and timestamp
FR-AML-002: Sanctions Screening
System shall screen customers and beneficiaries against:
UN Security Council Consolidated List
OFAC (US Office of Foreign Assets Control)
EU Sanctions List
UK HM Treasury List
Local country sanctions lists
System shall perform screening at:
Customer onboarding
Beneficiary addition
Transaction initiation
System shall generate alerts for matches above threshold

5.9 General Ledger Module
5.9.1 Business Requirements
Maintain accurate double-entry accounting records for all financial transactions, enabling financial reporting and audit compliance.
5.9.2 Functional Requirements
FR-GL-001: Chart of Accounts
System shall maintain master chart of accounts
System shall support account structure:
Account code (numeric or alphanumeric)
Account name
Account type (Asset, Liability, Equity, Revenue, Expense)
Account subtype
Parent account (for hierarchical structure)
Currency
Normal balance (Debit/Credit)
Active/inactive status
System shall allow creating new accounts with approval
System shall prevent deletion of accounts with transactions
System shall allow account hierarchy up to 5 levels
FR-GL-002: Automated Journal Entries
System shall automatically generate journal entries for:
Customer transactions (deposits, withdrawals, transfers)
Currency exchanges (realized gains/losses)
Fee collections
Bank transfers
System shall ensure all entries balance (debits = credits)
System shall generate unique journal entry reference number
FR-GL-003: Account Balances
System shall maintain real-time account balances
System shall calculate:
Opening balance for period
Debit total
Credit total
Closing balance
System shall display balances by:
Individual account
Account type/category
Currency
Accounting period
System shall support multi-currency consolidation

6. NON-FUNCTIONAL REQUIREMENTS
6.1 Performance Requirements
NFR-PER-001: Response Time
Web pages shall load within 3 seconds under normal conditions
Transaction processing shall complete within 5 seconds
Search queries shall return results within 2 seconds
Reports shall generate within 30 seconds for standard date ranges
NFR-PER-002: Throughput
System shall support minimum 500 concurrent users
System shall process minimum 1,000 transactions per hour
System shall handle peak loads 3x normal volume
NFR-PER-003: Scalability
System architecture shall support horizontal scaling
Database shall be optimized for growth to 1 million customers
System shall handle 1 billion transaction records
6.2 Security Requirements
NFR-SEC-001: Data Encryption
All data in transit shall be encrypted using TLS 1.3
All sensitive data at rest shall be encrypted using AES-256
Database connections shall be encrypted
Passwords shall be hashed using bcrypt or stronger algorithm
NFR-SEC-002: Authentication
System shall enforce strong password policy (minimum 12 characters, complexity requirements)
Multi-factor authentication shall be mandatory for all users
Sessions shall timeout after 15 minutes of inactivity
System shall lock accounts after 5 failed login attempts
NFR-SEC-003: Authorization
System shall implement role-based access control (RBAC)
Permissions shall be enforced at API level
Users shall have access only to authorized data and functions
All authorization decisions shall be logged
6.3 Availability and Reliability
NFR-AVL-001: Uptime
System shall maintain 99.9% uptime (maximum 8.76 hours downtime per year)
Planned maintenance shall be scheduled during off-peak hours
System shall provide advance notice for maintenance windows
NFR-AVL-002: Backup and Recovery
System shall perform automated daily backups
Recovery Point Objective (RPO): 24 hours
Recovery Time Objective (RTO): 4 hours
Backups shall be stored in geographically separate location
NFR-AVL-003: Disaster Recovery
System shall have documented disaster recovery plan
Disaster recovery procedures shall be tested annually
System shall support failover to backup site
6.4 Usability Requirements
NFR-USA-001: User Interface
Interface shall be intuitive and require minimal training
System shall follow consistent design patterns
System shall be responsive and work on desktop, tablet, and mobile
System shall be accessible (WCAG 2.1 Level AA compliance)
NFR-USA-002: Help and Support
System shall provide contextual help for complex features
System shall include user documentation and guides
System shall provide error messages that are clear and actionable
6.5 Compatibility Requirements
NFR-COM-001: Browser Support
System shall support latest versions of:
Google Chrome
Mozilla Firefox
Microsoft Edge
Safari
System shall gracefully degrade on older browser versions
NFR-COM-002: Integration Standards
System shall use industry-standard APIs (RESTful)
System shall support common data formats (JSON, XML, CSV)
System shall follow banking standards (ISO 20022, SWIFT, etc.)
6.6 Maintainability Requirements
NFR-MNT-001: Code Quality
Code shall follow established coding standards
Code shall be well-documented with inline comments
System shall have comprehensive unit and integration tests
Code coverage shall be minimum 80%
NFR-MNT-002: Modularity
System shall be built with modular architecture
Modules shall have well-defined interfaces
Changes to one module shall not impact others unnecessarily
NFR-MNT-003: Monitoring
System shall provide logging for troubleshooting
System shall monitor key metrics (response time, error rate, etc.)
System shall alert administrators on critical issues

14. RISKS AND MITIGATION
14.1 Technical Risks
Risk: Third-party API availability and reliability
Mitigation: Build retry mechanisms, implement caching, have fallback options, choose reliable vendors with SLAs
Risk: Data migration errors from existing system
Mitigation: Thorough testing of migration scripts, parallel run, data validation checksums, rollback plan
14.2 Project Risks
Risk: Scope creep extending timeline and budget
Mitigation: Clear requirements freeze date, change control process, prioritization of features, agile approach
